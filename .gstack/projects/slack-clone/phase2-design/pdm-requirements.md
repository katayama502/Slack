## 1. 機能要件一覧（MoSCoW法）

## 2. ユーザーストーリー（主要10件）

### US-01: 新規ユーザーのアカウント登録
**ユーザー**: 初めてアプリにアクセスするチームメンバー
> 「新しいチームメンバーとして、Googleアカウントで素早くログインしたい。なぜなら、アカウント情報を別途管理する手間なくすぐにチームのコミュニケーションに参加できるから。」

**受入条件**:
- Googleアカウントでワンクリックログインできる
- 初回ログイン時に表示名を設定できる
- ログイン後はメインのチャット画面に遷移する

---

### US-02: チャンネルへの参加
**ユーザー**: プロジェクトに新規参加したメンバー
> 「プロジェクトメンバーとして、既存のパブリックチャンネルを一覧から見つけて参加したい。なぜなら、チームの進捗やディスカッションに追いつけるから。」

**受入条件**:
- 全パブリックチャンネルの一覧を閲覧できる
- チャンネルの説明を見て参加を判断できる
- 参加後はサイドバーにチャンネルが追加される

---

### US-03: リアルタイムメッセージの送受信
**ユーザー**: 日常的にチームコミュニケーションを行うメンバー
> 「チームメンバーとして、テキストメッセージをリアルタイムで送受信したい。なぜなら、メール往復の遅延なくスピーディーに意思疎通できるから。」

**受入条件**:
- メッセージ送信後、即座に他のメンバーの画面に表示される
- ページリロードなしにメッセージが更新される
- 送信者のアバター・名前・送信時刻が表示される

---

### US-04: @メンションで特定メンバーに通知
**ユーザー**: 特定の人に連絡を取りたいメンバー
> 「メッセージ送信者として、`@`を入力して宛先ユーザーを指定したい。なぜなら、大勢がいるチャンネルでも特定の人に確実に内容を届けられるから。」

**受入条件**:
- `@` 入力後にユーザー名の補完候補が表示される
- メンションされたユーザーはハイライト表示で視認できる
- メンションされたユーザーに通知バッジが付く

---

### US-05: スレッドで話題を整理
**ユーザー**: 複数の話題が飛び交うアクティブなチャンネルの参加者
> 「チャンネル参加者として、既存のメッセージにスレッドで返信したい。なぜなら、メインのチャンネルを乱雑にすることなく関連する議論をまとめられるから。」

**受入条件**:
- メッセージにカーソルを当てると「返信」ボタンが表示される
- スレッドは右サイドパネルで開く
- スレッドへの返信数がメッセージ下部に表示される

---

### US-06: 絵文字リアクション
**ユーザー**: 簡潔にフィードバックを伝えたいメンバー
> 「受信者として、文章を打たずに絵文字でメッセージへの反応を示したい。なぜなら、了解・賛成・感謝などを素早く伝えてチャンネルのノイズを減らせるから。」

**受入条件**:
- メッセージホバー時に絵文字ピッカーが表示される
- 同じ絵文字を複数人が押すとカウントが累積表示される
- 自分が押したリアクションはハイライト表示され、再クリックで取り消せる

---

### US-07: メッセージの編集と削除
**ユーザー**: 誤送信・誤字に気づいたメンバー
> 「メッセージ送信者として、自分が送ったメッセージを編集または削除したい。なぜなら、誤った情報の拡散を防ぎ、正確なコミュニケーションを保てるから。」

**受入条件**:
- 自分のメッセージのみ編集・削除ボタンが表示される
- 編集後は「(編集済み)」ラベルが付く
- 削除後はメッセージが「このメッセージは削除されました」に置き換わる

---

### US-08: アバターとプロフィールの設定
**ユーザー**: チームに自分を紹介したい新規メンバー
> 「ユーザーとして、プロフィール画像と表示名を設定したい。なぜなら、テキストだけでなく顔と名前でチームメンバーに認識してもらい、コミュニケーションを人間的にできるから。」

**受入条件**:
- 画像ファイルをアップロードしてアバターに設定できる
- 表示名をいつでも変更できる
- 変更は即座に全チャンネルのメッセージ表示に反映される

---

### US-09: オンライン状態の確認
**ユーザー**: 今すぐ返事がもらえるか確認したいメンバー
> 「メッセージを送る前に、相手がオンラインかどうか確認したい。なぜなら、急ぎの案件で即レスが期待できる相手にだけメンションを送れるから。」

**受入条件**:
- サイドバーのユーザー一覧にオンライン状態がアイコンで表示される
- アクティブ（緑）・離席中（黄）・オフライン（グレー）の3状態
- アプリを閉じると一定時間後にオフライン状態に変わる

---

### US-10: チャンネルの作成
**ユーザー**: 新しいプロジェクトや話題を始めたいチームリーダー
> 「チームリーダーとして、特定の話題やプロジェクト専用のチャンネルを作成したい。なぜなら、関連する議論を一か所にまとめて情報の散乱を防げるから。」

**受入条件**:
- チャンネル名（英数字・ハイフン）と説明を入力して作成できる
- 作成後は自動で作成者がチャンネルに参加する
- 作成したチャンネルは即座に全ユーザーの「チャンネルを探す」一覧に表示される

## 4. 非機能要件

### 4.1 パフォーマンス目標

| 指標 | 目標値 | 計測方法 |
|------|--------|----------|
| 初回ページロード時間 (FCP) | 2秒以内 | Lighthouse / Core Web Vitals |
| メッセージ表示レイテンシ | 500ms以内 | Firestoreリアルタイムリスナー計測 |
| インタラクション応答 (INP) | 200ms以内 | Core Web Vitals |
| 同時接続ユーザー | 100ユーザー（MVP目標） | Firebase負荷テスト |
| メッセージ表示件数 | 最新50件をページング | 仮想スクロール検討 |

### 4.2 セキュリティ要件

| 要件 | 内容 | 実装方法 |
|------|------|----------|
| 認証必須 | 未認証ユーザーはチャット画面にアクセス不可 | Firebase Auth + Reactルートガード |
| データアクセス制御 | 参加チャンネルのデータのみ読み書き可能 | Firestoreセキュリティルール |
| メッセージ所有権 | 自分のメッセージのみ編集・削除可能 | Firestoreセキュリティルール（uid検証） |
| ファイルアップロード制限 | 最大10MB、画像・PDFのみ許可 | Firebase Storage セキュリティルール + クライアントバリデーション |
| XSS対策 | ユーザー入力のHTMLエスケープ | Reactのデフォルト動作（dangerouslySetInnerHTML不使用） |
| HTTPS強制 | 全通信をHTTPS化 | Netlifyデフォルト（Let's Encrypt） |
| 環境変数管理 | Firebase設定キーを環境変数で管理 | Netlify環境変数 / `.env.local`（gitignore） |

### 4.3 可用性・保守性

| 要件 | 内容 |
|------|------|
| ホスティング可用性 | Netlify SLA 99.99%に準拠 |
| オフライン対応 | Firebase offline persistence を有効化し、ネットワーク断時もキャッシュ表示 |
| エラーハンドリング | 送信失敗時のリトライUI・エラートースト通知 |
| レスポンシブ対応 | デスクトップ優先、タブレット（768px以上）で動作確認 |

### 4.4 アクセシビリティ

| 要件 | 内容 |
|------|------|
| キーボード操作 | Enterキーでメッセージ送信、Escでモーダル閉じる |
| フォーカス管理 | モーダル開閉時のフォーカストラップ |
| ARIAラベル | アイコンボタンへのaria-label付与 |
| カラーコントラスト | WCAG AA基準（4.5:1以上）を目標 |

## 6. スコープ外（MVP除外）

以下はMVPスコープ外とし、将来フェーズで検討する。

- 音声・ビデオ通話
- ハドル（音声チャンネル）
- Slackbot / Webhook 連携
- ワークフロービルダー
- エンタープライズ向けSAML/SSO認証
- メッセージの既読管理（詳細）
- モバイルアプリ（iOS/Android）
- 多言語対応（i18n）
