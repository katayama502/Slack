## 2. Firebase Firestore 読み書きコスト試算

### 前提条件

```
- 1日のメッセージ送信数: 平均50メッセージ/人
- メッセージ送信: 1回の書き込み = 1 write
- メッセージ受信 (リアルタイムリスナー):
    初回接続時: 全メッセージ読み込み (過去200件と仮定)
    差分更新: 1メッセージ受信 = 1 read
- チャンネル一覧取得: 1回/接続 = 1 read
- ユーザーリスト取得: 1回/接続 = 1 read  
- スレッド展開: 10 reads/スレッド展開
- メンション通知: 1 read/通知確認
```

### DAUごとの1日あたり操作数試算

#### DAU 100人の場合

| 操作 | 計算式 | 1日の回数 |
|------|--------|-----------|
| メッセージ書き込み | 100人 × 50件 | 5,000 writes |
| メッセージ読み取り (差分) | 100人 × 50件受信 | 5,000 reads |
| 初回接続読み取り | 100人 × 200件 | 20,000 reads |
| チャンネル一覧 | 100人 × 2回/日 | 200 reads |
| ユーザーリスト | 100人 × 2回/日 | 200 reads |
| スレッド展開 | 100人 × 5スレッド | 500 reads |
| **合計** | | **25,900 reads / 5,000 writes** |

**無料枠余裕**: reads 50,000 - 25,900 = **24,100 reads 余裕あり** ✅  
**無料枠余裕**: writes 20,000 - 5,000 = **15,000 writes 余裕あり** ✅  
**判定: Spark (無料) プランで対応可能**

## 4. コスト最適化提案

### 4-1. Firestoreクエリ最適化方針

#### ページネーション (最重要)
```
// 悪い例: 全件取得 (1チャンネル1,000件なら1,000 reads)
const snapshot = await getDocs(collection(db, "messages"));

// 良い例: 直近50件のみ取得 (50 reads)
const q = query(
  collection(db, "messages"),
  orderBy("createdAt", "desc"),
  limit(50)
);
```
**効果**: 初回接続時の reads を 75〜80% 削減

#### 複合インデックスの活用
```
// チャンネルID + タイムスタンプの複合インデックスを作成
// → チャンネル絞り込みが1クエリで完結し、クライアント側フィルタリング不要
Index: channels/{channelId}/messages - createdAt DESC
```

#### メッセージ数のカウンターフィールド
```
// 毎回 count() クエリを発行しない
// チャンネルドキュメントに messageCount フィールドを持たせる
// Cloud Functions (Blaze) か クライアントインクリメントで更新
```

### 4-2. キャッシュ戦略

#### Firestoreオフラインキャッシュの有効化
```javascript
// firebase.js
import { initializeFirestore, persistentLocalCache } from 'firebase/firestore';

const db = initializeFirestore(app, {
  localCache: persistentLocalCache()
});
```
**効果**: 再接続時の reads を最大60%削減。過去メッセージの再取得が不要になる。

#### React Query / SWR によるメモリキャッシュ
```
- メッセージリストを staleTime: 30秒 でキャッシュ
- ユーザーリスト・チャンネルリストは staleTime: 5分
- 同一セッション内での重複fetchを防止
```
**効果**: DAU 1,000人規模で推定 reads を20〜30%削減

#### リアルタイムリスナーの適切なデタッチ
```javascript
// コンポーネントのunmount時に必ずリスナーを解除
useEffect(() => {
  const unsubscribe = onSnapshot(q, (snapshot) => { ... });
  return () => unsubscribe(); // ← 必須
}, [channelId]);
```
**効果**: 不要なリスナーによる reads 課金を防止

### 4-3. 無料枠を最大活用する設計方針

#### Reads 削減の設計原則

| 設計方針 | 削減効果 | 実装難度 |
|----------|----------|----------|
| メッセージ取得を limit(50) に制限 | 大 | 低 |
| Firestoreオフラインキャッシュ有効化 | 大 | 低 |
| チャンネル切替時のみデータ再取得 | 中 | 低 |
| ユーザーリストをローカルStateに保持 | 中 | 低 |
| 画像・ファイルはStorageへオフロード | 中 | 中 |
| メッセージ集約 (バッチ書き込み) | 小 | 高 |

#### アーキテクチャ上の注意点

1. **Realtime Database vs Firestore**: チャット特化なら Realtime Database の方が接続コストが安いケースがあるが、クエリ柔軟性でFirestoreを推奨。
2. **Cloud Functions の使いどころ**: Spark プランでは利用不可のため、メンション通知はクライアントサイドで処理するか Blaze 移行後に実装する。
3. **Firebase Extensions**: Blaze プラン移行後に Trigger Email 等の拡張機能も低コストで追加可能。

### 結論

- **フェーズ0〜チーム内利用**: ランニングコスト完全無料で運用可能
